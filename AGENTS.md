# 注意

## 项目约束

- 本项目是无状态 DeepSeek 子代理 CLI，不是权限沙箱。调用方负责划定范围、评估风险和验证结果；不要在 CLI 内加入隐式授权或高风险自动操作。
- 使用 `Cargo.toml` 声明的最低 Rust 版本。调整最低版本或依赖后，同步更新文档、`Cargo.lock`、CI 和 release workflow；可复现的构建与测试使用 `--locked`。
- 修改 Rust 代码后至少运行：
  - `cargo +1.89.0 fmt --all -- --check`
  - `cargo +1.89.0 clippy --locked --all-targets --all-features -- -D warnings`
  - `cargo +1.89.0 test --locked --all-targets --all-features`
- `model-clis/homebrew-packages` 是本项目的下游包元数据仓库，维护 Homebrew Formula 和 Scoop manifest。发布新版本，或修改 release 资产名称、校验和及安装方式后，应在 release 完成后运行该仓库的 `Update packages` workflow，并确认生成文件校验与提交步骤成功。
- 保持工具接口小而明确。结果应为稳定、结构化且有界的 JSON；错误应包含足够的上下文和可执行的恢复提示。区分工具基础设施失败与被执行操作自身的失败状态。
- `fff-search` 使用固定 Git revision。每次 agent run 复用同一个搜索会话；初始化失败不得影响其他工具。搜索应遵守 workspace 和 ignore 边界，无效正则不得静默改变语义，查询、错误和结果输出必须有界。

## 提示词与模型行为

- 系统提示词和工具描述应简洁、中性，优先描述能力、参数和结果语义，不堆砌实现细节或重复 schema 已表达的信息。
- 除非正确性或安全边界确实要求，否则不要用强制路由提示限制模型选择工具。优先通过清晰 schema、无歧义且适度容错的参数语义、以及可操作的错误反馈减少调用错误。
- 对会影响模型工具选择、参数生成、错误恢复或任务完成方式的改动，除单元测试外，还应使用本地新编译的 CLI 调用真实 API 做行为测试。
- 行为测试使用真实、可验证且不泄露预期策略的任务描述。不要在任务中指定工具，除非测试目标本身就是指令遵循或特定工具行为。
- 根据改动风险设计一次性测试 workspace，并加入与目标行为相关的正常场景、边界条件或干扰项。记录工具序列、参数错误与恢复、答案正确性、完成状态和轮数；不要仅凭是否调用某个工具判断成功或失败。
- 根据测试中实际出现的稳定问题调整接口，不要为单次输出硬编码提示。修改后重新测试，并清理临时 workspace。
